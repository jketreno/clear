# [E] Ephemeral — Stop Editing Code You or AI Could Regenerate

> If a file can be regenerated from a source of truth, it should be regenerated — not patched. Patching generated code creates hidden debt: the next regeneration wipes your patch.

---

## The Problem

You have TypeScript types derived from Python models. The Python model changes. So you... edit the TypeScript file directly. Now you have hidden state. The next time someone runs the type generator, your edit disappears. Or worse — it silently persists because nobody runs the generator.

Or: you have React components generated from protobuf definitions. You add a special display case manually. The protos update. Someone regenerates. Your manual change is gone.

The solution: **fix the source, not the output**.

---

## What "Ephemeral" Means

Generated code is ephemeral: it has no independent existence. It is a rendering of something else. When you want to change what it renders, you change the source.

The same applies to boilerplate: code that follows a pattern so well-defined that AI could regenerate it correctly from the pattern alone.

**Ephemeral code = code that is safe to delete and regenerate at any time.**

---

## Identifying Generated Code

A file is generated (and should never be hand-edited) if:

1. It has a generation header:
   ```
   // @generated
   // DO NOT EDIT DIRECTLY
   // Generated by: [tool or skill name]
   // Source: [source file or system]
   ```

2. Its path is in a generated directory: `src/generated/`, `__generated__/`, `gen/`

3. Its path has `full-autonomy` in `clear/autonomy.yml` for a generation reason

## Setting Up a Regeneration Workflow

The key artifact is a **skill file** — a document that tells your AI exactly how to regenerate a category of code.

### Anatomy of a Skill File

```markdown
# Skill: [Category] Regeneration
# When to apply: [trigger conditions]
# Source of truth: [what to read]
# Output location: [where to write]

## Step-by-Step Process
1. [Read the source]
2. [Apply transformation rules]
3. [Write output with @generated header]
4. [Run verify-ci.sh to verify consistency]

## Transformation Rules
[Specific rules: type mappings, naming conventions, etc.]

## What NOT to Do
- Never hand-edit output files
- ...
```

See `clear/examples/skills/` for complete examples:
- `type-sync.md` — Python Pydantic → TypeScript interfaces + Zod schemas
- `proto-visualization.md` — Protobuf → React components
- `api-endpoint.md` — Feature request → endpoint with tests and docs

### Installing a Skill

**VS Code/Copilot:** Place in `.github/instructions/` with appropriate `applyTo` pattern:
```markdown
---
applyTo: "backend/models/api/**"
---
[skill content]
```

**Claude Code:** Reference from `CLAUDE.md` or create a custom command in `.claude/commands/`:
```markdown
# /project:sync-types
When asked to sync types, follow these rules: ...
```

**Cursor:** Create a `.cursor/rules/[skill-name].mdc` with appropriate `globs`:
```markdown
---
description: Type sync skill
globs: ["backend/models/**"]
---
[skill content]
```

## The Regeneration Pattern in Practice

### Python → TypeScript Sync

```
Source: backend/models/api/user.py (Python Pydantic model)
Output: frontend/src/types/api/user.ts (TypeScript interface)
        frontend/src/types/schemas/user.ts (Zod schema)
Skill:  clear/examples/skills/type-sync.md
```

**Trigger:** Python model changes  
**Command:** "Regenerate TypeScript types" or run `npm run gen:types`  
**Verify:** Architecture test `type-compatibility.test.ts` catches drift

### Proto → Components

```
Source: proto/services/*.proto (protobuf definitions)
Output: src/components/proto/**/*.tsx (React components)
        src/types/proto/*.ts (TypeScript types)
Skill:  clear/examples/skills/proto-visualization.md
```

**Trigger:** Proto files change  
**Command:** "Update the protos" or run `npm run gen:protos`  
**Verify:** Architecture test checks all messages have components

### OpenAPI → Mocks

```
Source: openapi.yml (API specification)
Output: tests/mocks/api/*.ts (mock handlers)
Skill:  [create your own based on clear/templates/skills/]
```

## Architecture Tests for Ephemeral Code

The Ephemeral principle works best when there's a test that detects when source and output have drifted:

```typescript
// tests/architecture/type-compatibility.test.ts
test('TypeScript types match Pydantic models', () => {
  // Load JSON schemas generated from Pydantic
  const schemas = loadSchemas('src/generated/schemas/');
  // Load TypeScript interfaces
  const interfaces = loadInterfaces('src/types/api/');
  
  // Every schema must have a corresponding interface
  for (const [name, schema] of schemas) {
    expect(interfaces.has(name)).toBe(true);
    
    // Required fields must appear in TypeScript
    for (const field of schema.required ?? []) {
      expect(interfaces.get(name)).toContain(toCamelCase(field));
    }
  }
});
```

When this test is in `verify-ci.sh`, the AI cannot finish a Python model change without also regenerating the TypeScript types.

## The `@generated` Contract

Add this header to every generated file. It signals to humans and AI that the file should not be hand-edited:

```typescript
// @generated — DO NOT EDIT DIRECTLY
// Source: backend/models/api/user.py
// Regenerate: npm run gen:types
// Generated at: [timestamp — optional, but useful for identifying staleness]
```

The architecture test `autonomy-guard.test.js` can be extended to:
1. Find all files with `@generated` headers
2. Verify they are in `full-autonomy` paths in `autonomy.yml`
3. Fail if a `@generated` file appears in a `supervised` or `humans-only` path (inconsistent boundary tagging)

## When Humans Should Edit "Generated" Code

Sometimes the generation template needs updating, not the output. The right sequence:

1. Identify that the generated output has a systematic problem
2. Find the **skill file** that produces it
3. Update the transformation rules in the skill file
4. Regenerate all outputs using the updated skill
5. Commit the updated skill + regenerated outputs together

This keeps the skill as the real source of architecture decisions.

---

## Measuring Success

You know [E] is working when:
- PRs touching generated files include updates to the corresponding source
- "I forgot to regenerate the types" stops being a PR comment
- Adding a field to a model takes one step, not three
- Generated code is confidently deleted during refactors because you know it can come back
